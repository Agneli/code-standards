Revisões de código

Uma revisão de código é a pedra angular do processo formal para garantir a qualidade da experiência do usuário desenvolvido pela equipe de Tecnologia Criativa. Trata-se de uma reunião entre os autores de marcação, revisores e outros interessados, com entrada esperado e as revisões de código subseqüentes. Basta colocar, nós o incentivamos a realização de revisões de código para manter as nossas ferramentas afiadas e limpas.

Por que eu deveria ter uma revisão de código?

As revisões de código são um investimento estratégico de tempo para mitigar o risco em um projeto.

Muitas vezes, os desenvolvedores de interface são convidados a marcação autor de quadros de arame ou composições visuais. No entanto, é possível que as telas que são projetados não pode ser traduzido para marcação com facilidade, ou sem comprometer a qualidade. As revisões de código fornecer uma oportunidade para o risco de ser tratadas e resolvidas antes que a produção total de páginas começa.

Revisão de código aumenta o nível global de conhecimento entre projetos

Desde que a revisão de código envolva membros de fora e de dentro da equipe, técnicas e melhores práticas são facilmente compartilhadas ao longo de toda a equipe de tecnologia criativa.

Revisão de código elimina bugs antes deles serem reproduzidos em alguns modelos e multiplicado em várias páginas

Idealmente, revisões de código são conduzidas no início do processo de desenvolvimento, antes da plena produção das páginas começarem. Quando os modelos são revisados pela equipe e executado através de ferramentas de validação de múltiplos navegadores, bugs devem começar a aparecer. Este é o momento ideal de serem corrigidos.

Revisão de código dá uma refrescada nos seus olhos e uma oportunidade de detectar os problemas com o código

Revisões de fora da equipe do projeto podem identificar mais fácilmente problemas com o código que os autores da marcação, que têm já tem trabalhado com o código por um longo período de tempo.

Quem deve participar da revisão de código?

Em última análise, o desenvolvedor líder é necessário para garantir que os prodecimentos de revisão adequados foram seguidos.

Idealmente, um líder de prática deve agir como facilitador para uma sessão de revisão de código, a menos que a liderança de prática é também o desenvolvedor de interface cujo código está em análise. Nessa situação, um gerente de projeto pode ser trazido como um facilitador.

A equipe de revisão deve incluir pelo menos dois membros sênior da equipe de tecnologia de Interface versados em práticas recomendadas e desenvolvimento.

O que é necessário para uma revisão de código?

Antes da revisão de código ser conduzida, modelos para serem revisados devem estar completamente desenvolvidos, validados e testado contra navegadores e plataformas necessárias para o projeto.

A líder prático e/ou desenvolvedor de interface deve distribuir em 48 horas anteriores à data da revisão de código:

  • Código para todas as páginas, includes associados do lado do servidor, CSS e JavaScript. Estes devem estar completamente comentados, impresso com os números de linha no lado esquerdo, e o nome do arquivo/página no rodapé de cada página impressa.
  • Imagens (screenshots) de cada modelo
  • URLs dos modelos, se for o caso
  • Uma lista com os navegadores e plataformas suportados por este projeto
  • Uma lista com conhecidos
  • Uma lista de problemas conhecidos e zonas de interesse

É comum para o código ser mudado constantemente até chegar a data da revisão do código. Infelizmente, isto não deixa tempo suficiente para validação e testes. Se for o caso, é recomendado que a revisão do código seja remarcada para uma data que garante uma revisão de código adequada.

Além disso, o líder na prática e/ou o desenvolvedor da interface deve reservar uma sala de conferência e chamada em número para todos os participantes, já que é possível que os membros do projeto ou equipe de revisão do código são off-site. Uma hora deve prever tempo suficiente para revisão de dois ou três modelos; embora, tempo variará dependendo do volume e complexidade dos modelos.

Durante a revisão do código, o líder na prática e/ou desenvolvedor da interface deve facilitar o encontro, enquanto o líder prático ou o gerenciador de projeto toma notas e atribui itens de ação. Os Comentadores deveriam ter revisto o código e vem preparado para fazer perguntas ou fornecer comentários.

Notas e ítens de ação (incluindo proprietários) deve ser distribuída a todos os participantes após a revisão de código. Se mudanças substanciais sair de uma revisão, ou todo o código não foi revisado, será necessário agendar uma segunda revisão dele. No entanto, isso deve ser discutido entre a equipe do projeto para determinar a viabilidade.
